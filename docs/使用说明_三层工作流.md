# 使用说明：三层工作流（脚本层 → 蓝图层 → 合成层）

本工具用于制作“点击交互式小故事”（选择/分支/结局），推荐按三层工作流逐步推进，避免结构与演出数据相互覆盖造成丢失。

对应项目文件（文件项目制）：

```
storage/projects/<projectId>/
  scripts.json        # 第一层：脚本层（分镜卡片）
  blueprint.json      # 第二层：蓝图层（结构/占位/分支）
  project.json        # 第三层：合成层（角色/资源）
  story.json          # 第三层：合成层（可运行故事）
  meta.json           # 元信息（含 aiFormula）
  assets/             # 项目素材
```

---

## 0. 工作台（Hub）

你可以在工作台：
- 新建故事草稿（空白）
- AI 生成新故事（脚本层草稿）
- 删除草稿

AI 生成时的“结构公式”（推荐使用）：
- 选择点数量：1~3
- 每个选择点选项数：2~5
- 结局数量：2~3

该公式会保存到项目 `meta.json.aiFormula`，并用于：
- AI 生成时强制输出格式
- 脚本→蓝图时按公式解析与校验（发现不一致会提示）

## 1. 第一层：脚本层（Script Studio）

**目的**：只写“发生了什么”，形成可读、可讨论、可迭代的分镜草稿。

脚本层你可以做：
- 新建/删除分镜卡片
- 调整顺序（上移/下移）
- 修改卡片名称与文本
- 选择并编辑某一张卡

脚本层你不应该做：
- 手工写蓝图字段（例如 `toNodeId`、节点 id、图结构）
- 试图在脚本层指定真实资源（图片/音频 id）

### 1.1 选择点与后果卡的推荐写法（重要）

为了让蓝图层能稳定解析并自动连线，推荐使用数字格式：

**选择点卡**：在 `text` 中换行写选项（必须换行）

```
选项1：……
选项2：……
（可选）选项3：……
```

**后果卡**：用 `name` 命名为 `i后果k`
- i = 第几个选择点（按出现顺序从 1 开始）
- k = 第几个选项（1..N）

示例：
- 第 1 个选择点的第 2 个选项后果：`1后果2`
- 第 2 个选择点的第 1 个选项后果：`2后果1`

**结局卡**：`name` 以 `结局` 开头，建议写成 `结局1`、`结局2`…

> 常见错误：把后果卡写成“3后果1”，把 i 当成“卡片序号”。i 实际是“第几个选择点”。

## 2. 第二层：蓝图层（Blueprint Studio）

**目的**：把脚本映射为“可执行结构”（节点/分支/占位），并为合成层提供明确的骨架。

蓝图层能解决的问题：
- 结构化分支：每个选择点变成可执行的 choices 跳转
- 分支收束：避免分支后果互相串线，统一收敛到后续节点/结局
- 校验：发现不可达节点、缺少跳转目标、没有可达结局等结构问题
- 占位绑定（P0）：为后续合成准备 Actor/Background/Event 的占位（不绑定真实资源）

### 2.1 从脚本进入蓝图时发生了什么？

点击“进入蓝图”会执行“脚本 → 蓝图编译”，典型处理包括：
- 识别选择点（`选项1/2/3…`）
- 识别后果卡（`i后果k`）
- 生成分支跳转与收束（避免后果卡串线）
- 校验可达性与结局（提示不可达节点、缺失后果等）

弹窗中的提示：
- **编译提示**：脚本格式/公式不匹配、缺少后果卡时的临时跳转等
- **校验提示**：不可达节点、场景无选项、无可达结局等
- **AI 解释原因（可选）**：基于“结构公式 + 编译提示 + 校验结果”给出人类可读解释与脚本修改建议（并可缓存）

### 2.2 “不可达结局/不可达节点”代表什么？

含义：从 `startNodeId` 出发，沿着 choices 跳转无法走到该节点。

常见原因（脚本层修正即可）：
- 写了 `结局2`，但没有任何分支/后果指向它（只是“顺序堆在最后”）
- 后果卡编号不对（应为 `i后果k`）
- 选择点缺少某个选项的后果卡（系统只能临时跳转收束）

## 3. 第三层：合成层（Compose Studio）

**目的**：在结构不变的前提下，把蓝图占位具体化为可运行、可导出的作品表现。

合成层你可以做：
- 绑定真实资源（图片/音频）
- 调整角色摆放（位置/缩放/旋转）
- 编辑节点的表现与时间线（根据当前 P0/P1 能力）
- 预览运行效果、导出 H5

合成层你不应该做：
- 改结构（增删场景/结局、改分支拓扑）

原因：结构应由蓝图层管理；合成层结构变动会导致脚本/蓝图/合成三层脱节。

---

## 4. 推荐工作方式（最省心）

1) 在脚本层把“选择点/后果/结局”写清楚（按约定格式）
2) 进入蓝图层，看编译/校验提示，把不可达/缺后果先解决
3) 在蓝图层完善分支与占位（Actor/Background/Event）
4) 进入合成层做演出与资源绑定，最终导出

当出现问题时：
- 优先回到脚本层修正文案/结构约定
- 其次在蓝图层调整结构（choices/收束）
- 不要在合成层改结构
